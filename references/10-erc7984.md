# ERC-7984 — confidential fungible tokens

ERC-7984 is OpenZeppelin's standard for confidential ERC-20-equivalent tokens. Balances are encrypted; transfer amounts are encrypted; only the balance holder can read their own amount (via user decryption).

## Package and imports

```bash
npm install @openzeppelin/confidential-contracts
```

```solidity
import { ERC7984 } from "@openzeppelin/confidential-contracts/token/ERC7984/ERC7984.sol";
import { FHE, externalEuint64, euint64 } from "@fhevm/solidity/lib/FHE.sol";
import { ZamaEthereumConfig } from "@fhevm/solidity/config/ZamaConfig.sol";
```

Current version: `^0.4.0` (released 2026-03-30).

## ERC-7984 vs ERC-20

| Property | ERC-20 | ERC-7984 |
|---|---|---|
| Balance visibility | Public | Encrypted (owner-only decryptable) |
| Transfer amount | Public in calldata | Encrypted handle + proof |
| Storage type | `uint256` | `euint64` |
| `balanceOf` | Returns `uint256` | Returns `bytes32` handle — decrypt off-chain |
| Approval pattern | `approve`/`transferFrom` | Operator pattern + signed permits |

**ERC-7984 balances are `euint64`.** That's a max of ~1.8 × 10^19 units. Pick your decimals accordingly (most tokens: 6 decimals → ~18 trillion whole tokens).

## Minimal token contract

```solidity
// SPDX-License-Identifier: BSD-3-Clause-Clear
pragma solidity ^0.8.27;

import { Ownable2Step, Ownable } from "@openzeppelin/contracts/access/Ownable2Step.sol";
import { FHE, externalEuint64, euint64 } from "@fhevm/solidity/lib/FHE.sol";
import { ZamaEthereumConfig } from "@fhevm/solidity/config/ZamaConfig.sol";
import { ERC7984 } from "@openzeppelin/confidential-contracts/token/ERC7984/ERC7984.sol";

contract MyConfidentialToken is ZamaEthereumConfig, ERC7984, Ownable2Step {
    constructor(
        address owner,
        uint64 initialSupply,
        string memory name_,
        string memory symbol_,
        string memory contractURI_
    )
        ERC7984(name_, symbol_, contractURI_)
        Ownable(owner)
    {
        euint64 encAmount = FHE.asEuint64(initialSupply);
        _mint(owner, encAmount);
    }
}
```

### What you get for free

- `confidentialBalanceOf(address)` returns an `euint64` handle for the account's balance.
- `confidentialTransfer(address to, bytes32 encAmount, bytes inputProof)` — the main user-facing transfer.
- `confidentialTransferFrom(address from, address to, bytes32 encAmount, bytes inputProof, ...)` — operator-initiated transfer.
- Internal `_mint(to, euint64)` and `_burn(from, euint64)` hooks.

## Visible vs confidential mint

ERC-7984 supports both public (fixed amount) and confidential (encrypted amount) minting. Pick based on whether the mint amount itself should be secret.

```solidity
// Public mint: amount is visible on-chain (encoded in calldata)
function mint(address to, uint64 amount) external onlyOwner {
    _mint(to, FHE.asEuint64(amount));
}

// Confidential mint: amount is encrypted
function confidentialMint(
    address to,
    externalEuint64 encAmount,
    bytes calldata inputProof
) external onlyOwner returns (euint64 transferred) {
    return _mint(to, FHE.fromExternal(encAmount, inputProof));
}
```

## Confidential burn

```solidity
function confidentialBurn(
    address from,
    externalEuint64 encAmount,
    bytes calldata inputProof
) external onlyOwner returns (euint64 transferred) {
    return _burn(from, FHE.fromExternal(encAmount, inputProof));
}
```

## Client-side transfer

```typescript
const TRANSFER_AMOUNT = 100n;

const input = instance.createEncryptedInput(
    await token.getAddress(),
    owner.address
);
input.add64(TRANSFER_AMOUNT);
const enc = await input.encrypt();

await token.connect(owner)[
    "confidentialTransfer(address,bytes32,bytes)"
](
    recipient.address,
    enc.handles[0],
    enc.inputProof
);
```

Note the explicit function-signature string — ERC-7984 has overloaded `confidentialTransfer` variants, so you need to disambiguate.

## Reading a balance on the frontend

```typescript
const handle = await token.confidentialBalanceOf(user.address);

const keypair = instance.generateKeypair();
// ... build EIP-712, sign, call userDecrypt (see references/07) ...
const balance = result[handle];
```

The ERC-7984 base contract **grants the balance holder ACL access by default** — you don't need to manually call `FHE.allow(balance, user)` inside the token. Inheriting contracts can still tighten or extend this.

## Wrapping ERC-20 → ERC-7984

Let users deposit public ERC-20 and receive confidential ERC-7984 of the same underlying asset. OpenZeppelin provides a wrapper base:

```solidity
import { ERC7984ERC20Wrapper } from "@openzeppelin/confidential-contracts/token/ERC7984/extensions/ERC7984ERC20Wrapper.sol";

contract MyUsdcConfidential is ZamaEthereumConfig, ERC7984ERC20Wrapper {
    constructor(IERC20 underlying_, string memory name_, string memory symbol_, string memory uri_)
        ERC7984(name_, symbol_, uri_)
        ERC7984ERC20Wrapper(underlying_)
    {}
}
```

`wrap(address to, uint256 amount)` pulls ERC-20 from the caller (via approval) and mints encrypted tokens to `to`.

Unwrapping is asynchronous: the user burns encrypted tokens, the contract requests public decryption of the amount, and upon proof callback the underlying ERC-20 is released. This is the canonical two-step decryption pattern from `references/06-decryption.md`.

## Vesting wallet

OpenZeppelin also provides `ConfidentialVestingWallet` for time-locked distribution of ERC-7984 tokens:

```solidity
import { ConfidentialVestingWallet } from "@openzeppelin/confidential-contracts/finance/ConfidentialVestingWallet.sol";
```

Same API as the regular OZ vesting wallet, but the released amounts are encrypted transfers.

## Anti-patterns

- **Using `uint256` for balances.** ERC-7984 is `euint64`. Design for 64-bit amounts.
- **Forgetting the `ZamaEthereumConfig` inheritance.** Even with ERC-7984 inherited, you still need the network config base.
- **Calling `balanceOf`-like functions and expecting a plaintext.** `confidentialBalanceOf` returns a handle; decrypt client-side.
- **Skipping the function-signature disambiguation on overloaded calls.** Ethers needs the explicit selector string for overloaded `confidentialTransfer`.
- **Trying to unwrap synchronously.** Unwrapping requires public decryption and therefore a two-step flow.
